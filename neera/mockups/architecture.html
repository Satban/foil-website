Nishi Architecture

Phased build-out · MVP through Year 2+

The architecture below is designed for evolution. Each phase adds capability without throwing away the previous foundation. Phase 1 ships fast; the seams for internal module decomposition, multi-tool, and multi-tenant are already in place but not operationalized until they're needed. Core thesis: Nishi is positioned in the observed world, not the stated world. Most nutrition apps live on what users report (logged meals, claimed goals, chat statements). Nishi lives on what users actually do — receipts (what they bought), wearables (how the body responded), adherence (did they follow through). The stated-to-observed gap is where every health-app retention curve dies; closing it is the moat.

Architectural principle (locked in 2026-06-04): single voice externally, modules internally. Reject multi-agent orchestration as user-visible architecture until forced by scale. The user never knows there are five modules under the hood; they just talk to Nishi.

Day-1 Architectural Principles

These are the choices made in Phase 1 that compound (or constrain) every later phase.

Provider-agnostic LLM layer

Wrap every LLM call behind an interface. Don't hard-code OpenAI. 30 min on day 1, saves 3 weeks at month 18.

Tool use as first-class

Every external API is registered as a tool. Add new tools by registering them; the agent loop doesn't change.

Four-layer memory

Short-term session + rolling behavioral patterns + long-term profile + raw archive. All in Supabase (Postgres + pgvector + Storage). Pattern re-derivation runs weekly. Rolling behavioral is the moat.

Python AI service separate

Even with one agent, deploy the AI orchestration as its own service. Agents inside Supabase edge functions break by Phase 2.

Field-level encryption on health-sensitive data

Medications, conditions, lab values encrypted at column level. Doesn't slow MVP; retrofitting is brutal.

Eventing infrastructure

Proactive nudges are 50%+ of agent value. Build the event bus in Phase 1 even with 3 events.

LLM telemetry from day 1

Helicone or Langfuse on every LLM call. You can't optimize what you don't measure.

Explainable memory

Every memory-driven recommendation can surface its "why" (source, date, confidence). Keeps Nishi on the right side of the useful-vs-creepy line.

Cancel ≠ delete

Subscription lifecycle preserves memory at ~$0.05/mo for cancelled users (30% return within 90 days). Hard delete only on user request. HIPAA-eligible data exempt from time-based deletion.

Hand-rolled orchestration

~50 lines of Python for Phase 1 and Phase 2. No framework. Adopt LangGraph / Agents SDK only when real pain forces it — probably Phase 3, possibly never.

Structural readiness without complexity

Stub seams for multi-agent, multi-tenant, hardware ops. Don't operationalize until needed. Seams are cheap; retrofitting is not.
